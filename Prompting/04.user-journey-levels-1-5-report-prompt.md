# Level 1: Epic — Business goal

## Epic

**Build an invariant-based thinking training system**

---

## Purpose

Use the **4×4 Magic Square** problem as the vehicle to:

- Train **invariant-centered** design thinking (express and protect domain truths before implementation details).
- Train **Dual-Track TDD** (domain track and contract/IO track in parallel, with clear separation).
- Train **explicit input/output contracts** (what is allowed in, what is guaranteed out, and how failures are represented).
- **Internalize** the loop: **design → test → implement → refactor**, with tests driving discovery of invariants and contracts.

---

| Area | Criterion |
|------|-----------|
| **Domain logic** | **≥ 95%** test coverage on entity/domain code. |
| **Input validation** | **100%** of contract tests for inputs pass (every stated rule has a failing test that becomes green). |
| **Implementation quality** | **No hardcoding** of problem-specific values in scattered logic; use named constants, types, or configuration at the right layer. |
| **Clarity** | **No magic numbers** (or unexplained literals); intent is visible in names and structure. |
| **Traceability** | Every important **invariant** is **mapped to at least one test** (Invariant → Test traceability for review and learning). |

# Level 2: User journey

## Persona

**Software development learner**

- Practicing **TDD** (small steps, failing tests first, refactor with safety).
- Learning **Clean Architecture** (clear boundaries, dependency direction, domain isolated from I/O).

---

## Journey

### Step 1 — Problem awareness

**Reframe the work:** this is not “implement a magic square” as a one-shot coding exercise.

The learner recognizes that the real goal is to **discover and protect invariants** (what must always be true), **define behavior through tests**, and **separate** “what the problem *is*” from “how we *talk* to it” (CLI/UI/API). Awareness sets up contract-first and domain-first work instead of jumping straight to a grid and loops.

---

### Step 2 — Contract definition

Before deep implementation, the learner **locks the surface**:

| Concern | What to define |
|--------|----------------|
| **Input schema** | Allowed types, ranges, structure (e.g. grid size, cell domain, partial vs complete square), normalization rules. |
| **Output schema** | Success shape (e.g. completed grid, diagnostics), empty vs impossible cases, stable ordering if relevant. |
| **Exception policy** | Which inputs are **errors** vs **valid but unsatisfiable**; which exceptions or result types cross the boundary; no silent swallowing of programmer mistakes. |

Contracts become **the second track** of TDD: red tests for invalid input, missing fields, and boundary values before the domain is “finished.”

---

### Step 3 — Domain separation

The learner splits the **entity / use-case** space into focused responsibilities (names can map to types or modules per your ECB layout):

| Piece | Role (conceptual) |
|-------|-------------------|
| **BlankFinder** | Locates empty cells / placeholders in a partial square. |
| **MissingNumberFinder** | Determines which value(s) are absent or deducible from constraints. |
| **MagicSquareValidator** | Checks magic-square rules on a given grid (rows, columns, diagonals, uniqueness if required). |
| **Solver** | Orchestrates search or deduction using the above (pure strategy, no UI). |

Separation supports **single responsibility**, easier **invariant → test** mapping, and **refactor** without rewriting everything.

---

### Step 4 — Dual-track progress

The learner advances **two tracks in parallel**, then integrates:

| Phase | UI / boundary track | Logic / domain track |
|-------|---------------------|----------------------|
| **RED** | Failing tests for parsing, validation at the edge, error messages, DTO mapping. | Failing tests for validators, finders, solver steps. |
| **GREEN** | Minimal code to satisfy boundary tests (thin adapters). | Minimal domain code to satisfy entity/control tests. |
| **REFACTOR** | **Integration cleanup:** shared types, duplication removal, naming, dependency direction (boundary → control → entity), no domain logic leaking into I/O. |

“Parallel RED” avoids a **thick UI** that encodes rules the domain should own, and avoids a **perfect domain** with no story for bad inputs.

---

### Step 5 — Regression protection

With core paths green, the learner **hardens** the system:

- **Edge cases:** boundary sizes, single blank, full grid, symmetry or duplicate constraints if applicable.
- **Input error cases:** malformed payloads, out-of-range values, wrong length, type mismatches—aligned with Step 2 contracts.
- **Combination failure cases:** valid-looking pieces that **together** violate an invariant or cannot be solved; ensures the solver and validator interact correctly under stress.

Each addition follows **red → green → refactor**, preserving **traceability** from invariant or contract rule to test.

---

This journey is how the persona reaches the Epic outcomes: **invariant-centered design**, **Dual-Track TDD**, **clear contracts**, and the **design → test → implement → refactor** habit—with **95%+ domain coverage**, **100% contract tests** on validation, **no hardcoding / magic numbers**, and **Invariant → Test** traceability.

# Level 3: User stories

---

## Story 1 — Input validation

**As a** learner  
**I want** input validated as exactly **4×4**  
**So that** invalid data never reaches the domain

### Acceptance criteria

- **Dimensions:** If the structure is not exactly **4×4**, raise an **exception** (or project-standard error type) at the boundary/validation layer.
- **Blanks:** If there are not **exactly two** blanks (per agreed blank representation), raise an **exception**.
- **Uniqueness:** If any **duplicate** filled values appear (among non-blank cells), raise an **exception**.
- **Range:** If any filled cell is **outside 1–16** (or violates the agreed numeric domain), raise an **exception**.

---

## Story 2 — Blank detection

**As a** learner  
**I want** to find **blank coordinates** accurately  
**So that** combination / placement attempts are possible

### Acceptance criteria

- **Ordering:** Coordinates are produced in **row-major** order (e.g. `(row, col)` sorted by row, then column).
- **Cardinality:** Exactly **two** coordinate pairs are returned when input satisfies Story 1.

*(Note: “0 coordinates” in your draft is treated as **blank** coordinates, consistent with two blanks and combination attempts.)*

---

## Story 3 — Missing number detection

**As a** learner  
**I want** the two absent values from **1–16** identified  
**So that** placement candidates are well-defined

### Acceptance criteria

- Given a valid partial square (Story 1) and blank positions (Story 2), the two **missing** numbers from **1–16** are computed correctly.
- The two missing values are returned in **ascending** order.

---

## Story 4 — Magic square judgment

**As a** learner  
**I want** a clear judgment of whether a **4×4** grid is a valid magic square  
**So that** the solver and tests share one definition of “magic”

### Acceptance criteria

- **Rows:** All **four** row sums are **equal**.
- **Columns:** All **four** column sums are **equal** and match the row sum.
- **Diagonals:** **Main** and **anti** diagonal sums equal that same magic sum.

---

## Story 5 — Two placement attempts

**As a** learner  
**I want** two ordered placement strategies over the blanks  
**So that** a small answer trace exists without full search complexity

### Acceptance criteria

- **First attempt:** Assign missing numbers in **ascending** order to blanks in **row-major** order (small value → **first** blank in that order).
- **Second attempt (on failure):** If the first ordering does not yield a valid magic square (Story 4), try the **reverse** assignment (e.g. ascending values paired to blanks in reverse row-major order, or the reverse pairing your spec defines—**document one rule** in tests).
- **Trace length:** The **answer** structure has **array length 6** (per your contract: e.g. six integers encoding placements + outcome flags—**define the exact layout** in contract tests so “6” is not a magic number in code).
