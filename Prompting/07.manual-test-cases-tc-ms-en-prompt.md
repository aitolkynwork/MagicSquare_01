The README spells out **task order and TDD**, not Git. Here is a simple branch model that fits that work.

## Default picture

- **`main`**: always matches what is on the remote (or what you consider “released / integrated”). Only merge finished, tested work here.
- **Short-lived branches**: one branch per **logical chunk** of work—not necessarily every micro sub-task (`TASK-001-1`, etc.), but often per **TASK-00x** or per **user story slice** (e.g. all of US-002 entity work).

That matches the README’s **vertical slice** idea (entity before control/boundary) without drowning in dozens of branches.

## Practical patterns

**1. Start from current `main`**

```powershell
git checkout main
git pull origin main
```

**2. Create a branch for the next slice**

Name branches so you can grep history later, e.g.:

- `task/001-magic-constant`
- `feat/us-002-partial-board-validation`
- `task/006-007-blanks-missing`

```powershell
git checkout -b task/001-magic-constant
```

**3. Commit in small steps (aligned with RED → GREEN → REFACTOR)**

- After a **green** step (or a coherent RED+GREEN pair), commit with a message that references the task ID, e.g. `TASK-001-2: define MAGIC_SUM_ORDER_4`.
- Keeps bisect and review easy; matches the README checkpoint mindset.

**4. Stay up to date if `main` moves**

```powershell
git fetch origin
git rebase origin/main
# or: git merge origin/main
```

Rebase = linear history; merge = preserves exact merge points. Solo projects often use rebase on feature branches.

**5. Finish the slice**

- Run `pytest` (and any coverage gate before TASK-027).
- Merge to `main`:

```powershell
git checkout main
git pull origin main
git merge task/001-magic-constant
git push origin main
```

Or open a **PR** from `task/...` → `main` if you use GitHub/GitLab (same idea, review in the UI).

**6. Delete the branch after merge**

```powershell
git branch -d task/001-magic-constant
git push origin --delete task/001-magic-constant   # if you pushed the branch
```

## How this maps to your checklist

| README idea | Git habit |
|-------------|-----------|
| Order `TASK-000` → … → `TASK-028` | New branch when you start a **new TASK** or **new US** block; merge when that block’s tests are green. |
| Checkpoints (e.g. TASK-005, TASK-014) | Good **merge** points: merge to `main` so `main` always reflects a stable checkpoint. |
| Long US-004 block (TASK-008–014) | Either **one** branch `feat/us-004-domain-completion` with many commits, or split at TASK-012 if that PR gets too large. |

## Commands you will use often

| Goal | Command |
|------|---------|
| See branches | `git branch -a` |
| See relationship to remote | `git status -sb` |
| Compare before push | `git log main..HEAD` |
| Abandon branch | `git checkout main` then `git branch -D branch-name` |

## Optional: one branch per TASK-ID

Fine if you like very small PRs; overhead is more branch switching. For **TASK-000-1 / -2 / -3**, keep them on **one** branch unless you need to land fixtures before other work.

---

**Summary:** Keep **`main` clean and integrated**; use **named short-lived branches** per task or per story slice; **commit after each meaningful TDD step**; **merge (or PR) at README checkpoints**. The README does not require a specific branch policy—this is a sensible default for solo or small-team work on that epic.
