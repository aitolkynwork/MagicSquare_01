## Level 4 — Implementation scenario (technical)

**Feature:** Complete 4×4 magic square  

**Actor:** Developer practicing TDD, verifying invariant-based logic.

### Background (shared)

- 4×4 matrix; **0** means blank.
- Exactly **two** cells contain **0**.
- Filled numbers in **1..16**; no duplicate non-zero values.
- **4×4 magic constant = 34** (must be a named constant in code).

### Scenarios

1. **Magic square completes with smaller→larger order** — Example matrix with zeros at (3,3) and (4,4) in 1-based terms; find blanks and missing numbers; place smaller in first blank (row-major), larger in second; assert all sums **34**, result length **6**, coordinates **1-indexed**.
2. **Reverse placement** — Same starting matrix; when primary assignment is not magic and reverse assignment is magic, return reverse result; final grid satisfies **34**.
3. **Wrong number of blanks** — Validation → error.
4. **Duplicate non-zero numbers** — Validation → error.
5. **Value out of range (e.g. > 16)** — Validation → error.

### Technical bindings

| Item | Rule |
|------|------|
| Blank | `0` |
| Magic sum | **34** |
| Blank order | Row-major for “first” / “second” blank |
| Output | Length-**6** array; document field order in one contract test |
| Coordinates | **1-indexed** in output |

### ECB note

- Validation errors: boundary/control per project choice; domain receives valid structures or raises domain errors—**one** policy, tested consistently.
- Pure magic checks and placement logic: **entity**; orchestration of try A then B: **control**.
