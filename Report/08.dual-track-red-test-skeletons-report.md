# Dual-track RED test skeletons (pytest + JUnit 5) — Report 08

**Scope of this document:** Executable **RED-only** test stubs for **dual-track TDD** — **Track A** (UI / boundary contract) and **Track B** (domain logic: `find_blank_coords`, `find_not_exist_nums`, `is_magic_square`, `solution`). Each test method contains a **single** `pytest.fail(...)` (Python) or `fail(...)` (JUnit 5); **no** production implementation, **no** GREEN, **no** REFACTOR in this deliverable.

**Date:** 2026-04-28 (initial).

**Authoritative contracts & invariants:** [Report/02 — Layer design, contracts, TDD](02.layer-design-contracts-tdd-report.md) (**D1–D7**, **T-U\***, **T-D\***, error literals).

**Related:** [Report/07 — Manual test cases TC-MS (English)](07.manual-test-cases-tc-ms-en-report.md) and [docs/test-cases-TC-MS-english.md](../docs/test-cases-TC-MS-english.md) (TC-MS groups A–D). Report/07’s deliverables table referenced a **`tests/tc_ms/`** layout; this repository’s **contract-first RED** stubs for the dual-track plan live under **`tests/ui_red/`** and **`tests/logic_red/`** as described below.

---

## 1. Task intent

1. Freeze **parallel** RED entry points so **UI** and **logic** tracks can fail independently before shared implementation.
2. Keep **one failing statement per test** so CI and local runs unambiguously show RED until real assertions replace the stub.
3. Align naming with **UI-RED-01…06** and **LOGIC-RED-01…04** (and subcases **03a–e**, **04a–c**) for traceability to design chat and Report/02 invariants.

## 5. How to run

**pytest** (from repository root; requires optional dev deps):

```text
pip install -e ".[dev]"
pytest tests/ui_red tests/logic_red -q
```

All tests are expected to **fail** until RED stubs are replaced with real calls and assertions.

**JUnit 5** (requires JDK 17 and Maven on `PATH`):

```text
cd test-skeletons/junit5-tc-ms
mvn test
```
