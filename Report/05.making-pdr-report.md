## 2. Dual-track UI + logic TDD with MLOps (concept agreed in discussion)

### 2.1 Core idea

- **UI track (boundary):** Verification driven by a **UX contract**—what is visible, possible, activated, included (e.g. no inappropriate error before input; named error when a named failure applies). Initial failing tests (**RED**) reflect design and observable behavior.
- **Logic track (domain):** Verification driven by **logic rules**—allow/reject, maintain/suspend, return/block, calculate/save at the rule level. **RED** reflects invariants and domain decisions (e.g. invalid completed grid).
- **MLOps (when ML is on the path):** Extends “named outcomes” to **data validity**, **versioned model behavior**, **evaluation and promotion gates**, and **reproducibility** (lineage, thresholds as **policy**). At problem-definition depth this stays **contract surfaces**, not a prescription for specific pipelines or frameworks.

### 2.2 TDD rhythm (philosophy)

*UI creates RED from design (UX contract); logic—and ML gates where relevant—creates RED from rules and evidence.* **GREEN** and **REFACTOR** are shared: avoid collapsing boundary concerns into domain tests or the reverse.

### 2.3 Controlled vocabularies (for scenario → test traceability)

| UX contract language (observable) | Logic rule language (decision) |
|-----------------------------------|--------------------------------|
| Visible / not visible | Allow / reject |
| Possible / impossible | Maintain / suspend |
| Activated / deactivated | Return / block |
| Included / not included | Calculate / save |

**Discipline:** Prefer scenarios that express a **decision** or **state change** when converting work items into tests; vague to-dos are poor test seeds.

### 2.4 Fit with this repository

Dual-track thinking aligns with Report/02’s separation of **boundary (screen)** vs **entity/control (domain)** and with traceability matrices. **MLOps** does not replace the logic track; it **gates and evidences** the ML-related part of that track while the UI track still proves observable outcomes for loading, errors, low-confidence, or fallback states when product policy requires them.

### 2.5 Risks called out in discussion

- ML is partly **statistical**: pair hard contracts (schemas, invariants) with **policy thresholds** and reproducible evals to avoid flaky or illusory GREEN.
- Keep product-visible thresholds and outcomes in the **mapping** between UX contract and logic/ML gates so retrains do not silently desync the UI story.

## 3. Organization plan (before editing `docs/PDR.md`)

Two scope strategies were identified for combining PDR purity with dual-track + MLOps:

| Strategy | Meaning |
|----------|--------|
| **A — PDR stays mostly pure** | Keep §§1–5 as the problem spine from Report/01; add a short bridge plus tables only. |
| **B — Widen PDR charter** | Explicitly widen the scope paragraph and reorganize the full body under UI / Logic / MLOps pillars (closer overlap with Report/02). |

**Chosen approach (implemented):** Extend scope slightly to include **named contract surfaces** (UX, logic, data/model policy when ML applies) while still excluding implementation stacks and algorithms. **Do not** relocate the full §§1–5 narrative: keep it intact under original headings so Report/01 alignment remains obvious; **insert** reading frame, cross-track mapping, MLOps lens, scenario mapping, and expand the document map and chat summary—i.e. a hybrid close to **A** with explicit dual-track + MLOps scaffolding.
